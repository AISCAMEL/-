## 2. システム構成・技術スタック

| レイヤー | 採用技術 | 役割 |
|---------|---------|------|
| フロントエンド | Next.js (App Router) + TypeScript | 公開面の表現力と管理面の実務UIを両立 |
| スタイル | Tailwind CSS | デザイントークン（後述）で世界観を統一 |
| 認証 | Supabase Auth | メール/パスワード + LINE Login（段階導入） |
| データベース | Supabase (PostgreSQL) | RLSで権限制御をDB層でも担保 |
| ストレージ | Supabase Storage | 投稿画像・アバター・商品画像 |
| 決済（Phase2） | Stripe 等 PSP | カード情報を自社保持しない |
| 外部導線 | LINE公式アカウント | 全面で強いCTAとして維持 |
| ホスティング | Vercel 想定 | Next.js最適 |

### 構成方針

- **モバイルファースト**（スマホ前提でUI最適化）。
- 権限制御は **UI層 + API層 + RLS の三重化**（フロントだけで隠さない）。
- 公開面（感性寄り）と管理面（運用寄り）でUIトーンを分けるが、配色基盤は共有。
- 次フェーズ拡張を阻害しない疎結合な構造（テーブル・コンポーネント単位）。

## 4. サイト全体構成

```
IWASAWA SURF BASE
├─ A. トップページ（入口 / Marketing）
├─ B. オンラインショップ（無在庫・受注起点 / 法務必須）
├─ C. コミュニティ（★会員制化）

横断基盤：Auth / Member / Role / LINE導線
```

### 4-A. トップページ

役割：サービス全体の入口。価値を一目で伝える。

| セクション | 内容 |
|-----------|------|
| Hero | フルスクリーンの海・波・透明感。キーメッセージ |
| Concept | ブランドメッセージ「福島の波を、もっと近くに。」 |
| 4サービス | スクール / レンタル / 移動サポート / ローカルガイド |
| Plan | 利用プラン |
| Area | 岩沢海岸・広野町の紹介 |
| Usage Flow | 利用の流れ（学ぶ→借りる→移動→案内→仲間） |
| About Us | 私たちについて（復興と再接続のストーリー） |
| FAQ | よくある質問 |
| Contact | 予約・問い合わせ |
| 導線バナー | Shop / Community への誘導、LINE CTA |

表現方針：ネイビー/オーシャンブルー/サンドベージュ、波や潮の流れの演出、スクロールで自然につながる体験。

### 4-B. オンラインショップ（Phase2、定義のみ本書に先出し）

役割：物販以上に「現地体験へつながるギア接点」「ブランド接点」を担う。

- 無在庫・受注起点型。小物→アパレル→ウェット→ボード周辺の順に拡大。
- **即納表現は禁止。納期は具体的に記載。** 「在庫あり」と誤認させない。
- 決済はPSP前提（カード情報を自社保持しない）。
- 必須要素：商品一覧 / カテゴリフィルター / カート / 購入・問い合わせ導線 / 特商法表記 / 返品・交換ポリシー / 発送目安 / 信頼訴求。

無在庫運用フロー：`注文 → 仕入先在庫確認 → 確保できた場合のみ受注確定 → 発送手配 → 発送通知 → 欠品時は代替提案 or 返金`。
→ 法務要件は「セクション9」、テーブルは「セクション7（Shop系）」参照。

### 4-C. コミュニティ（本リリースの主対象）

役割：地元の海を知る人・これから始める人・県外から訪れる人を自然につなぐ場。
既存構成（Hero / コンセプト帯 / 投稿フィード / カテゴリフィルター / 投稿フォーム / 波情報ウィジェット / イベント一覧 / メンバータイプ紹介 / 利用者の声 / LINE CTA）を**土台として維持**し、認証ゲートとマイページを重ねる。

カテゴリ：`waves` / `experiences` / `questions` / `events` / `gear`

### 4-D. 管理画面（本リリースの主対象）

役割：コミュニティを「運営できる状態」にする。装飾過多を避け、実務的で見やすく。

| AD-03 | イベント管理 | `/admin/events` | 作成・編集・受付状態切替・定員管理・表示順変更 |
| AD-04 | 会員管理 | `/admin/members` | 種別変更・状態(有効/停止)・管理者権限付与・投稿履歴確認 |

| AD-07 | 導線管理 | `/admin/navigation` | TOP/Shop/Community のバナー・CTA管理 |

### AD-01 ダッシュボード指標

- 会員数 / 新規登録数（当日・期間）
- 今日の投稿数
- イベント数（開催予定・受付中）
- 承認待ち件数（通報・要確認）
- よく見られているカテゴリ（閲覧数ランキング）

### 運用ルール

- 破壊的操作（削除・停止）は確認ダイアログ必須。
- 主要操作は `admin_audit_logs` に最小記録（誰が・何を・対象）。
- 一覧は検索・フィルタ・ページング前提。スマホでも最低限操作可。

## 7. データベース設計（テーブル定義）

> Supabase(PostgreSQL)想定。全テーブル共通で `id uuid PK`, `created_at timestamptz`, `updated_at timestamptz` を持つ。enum はアプリ定義 or PG enum。

### ① members（会員）
| カラム | 型 | 説明 |
|--------|----|------|
| id | uuid | PK（auth.users と1:1） |
| email | text(unique) | ログインID |
| handle | text(unique) | 表示用ハンドル |
| display_name | text | 表示名 |
| bio | text | 自己紹介 |
| avatar_url | text | アイコン |
| role | enum | visitor/beginner/local/staff/admin |
| status | enum | active/suspended |
| line_user_id | text | LINE連携(任意) |
| home_area | text | 地元/県外等の属性(任意) |
| last_login_at | timestamptz | 最終ログイン |

> パスワードは Supabase Auth が管理。本テーブルに生PW/ハッシュは保持しない。

### ② posts（投稿）
| カラム | 型 | 説明 |
|--------|----|------|
| id | uuid | PK |
| author_id | uuid | FK→members |
| category | enum | waves/experiences/questions/events/gear |
| title | text | 任意 |
| body | text | 本文 |
| status | enum | draft/published/hidden |
| is_featured | bool | 注目表示 |
| view_count | int | 閲覧数(人気カテゴリ集計) |
| like_count | int | 非正規化キャッシュ |

### ③ post_images（投稿画像）
| id | uuid PK / post_id uuid FK→posts / url text / sort_order int |

### ④ comments（コメント）
| id | uuid PK / post_id FK→posts / author_id FK→members / body text / status enum(published/hidden) |

### ⑤ likes（いいね）
| id uuid PK / member_id FK→members / post_id FK→posts / **unique(member_id, post_id)** |

### ⑥ events（イベント）
| カラム | 型 | 説明 |
|--------|----|------|
| id | uuid | PK |
| title / description / location | text | 基本情報 |
| starts_at / ends_at | timestamptz | 日程 |
| capacity | int | 定員 |
| entry_status | enum | upcoming/open/closed/finished |
| sort_order | int | 表示順 |
| created_by | uuid | FK→members(staff) |

### ⑦ event_entries（イベント申込）
| id uuid PK / event_id FK→events / member_id FK→members / status enum(applied/confirmed/cancelled/attended) / **unique(event_id, member_id)** |

### ⑧ reports（通報/要確認）
| id uuid PK / target_type enum(post/comment) / target_id uuid / reporter_id FK→members / reason text / status enum(open/reviewing/resolved/dismissed) / handled_by FK→members(staff) |

### ⑨ announcements（お知らせ）
| id uuid PK / title text / body text / placement enum(top/shop/community/all) / is_published bool / published_at timestamptz |

### ⑩ navigation_items（導線管理）
| id uuid PK / surface enum(top/shop/community) / label text / href text / kind enum(banner/cta/link) / is_active bool / sort_order int |

### ⑪ admin_audit_logs（操作ログ・最小）
| id uuid PK / actor_id FK→members(staff) / action text(例 post.hide) / target_type text / target_id uuid / meta jsonb |

### ⑫ waves_reports（波情報・Local投稿/任意）
| id uuid PK / author_id FK→members(local+) / observed_at timestamptz / wave_height text / wind text / condition text / spot text |

### Shop系（Phase2・定義のみ先出し、既存法務思想を踏襲）
- `products`：name / description / category / price / shipping_fee / lead_time_text(納期具体) / status（即納表現禁止）
- `product_variants`：size / color / sku
- `orders`：status `pending→stock_check→confirmed→shipped→delivered`（/ cancelled / refunded）。無在庫フロー準拠。**PSP参照IDのみ保持**
- `order_items` / `shipments` / `legal_pages`（特商法・返品・発送目安）

### RLS方針（要点）
- `posts`：published はゲスト読取可（ただしアプリ層でゲストは最新数件＋ぼかし）、draft/hidden は author か staff のみ。
- `likes`/`comments`：作成は認証済み(beginner+)のみ。
- `waves_reports`：作成は role≥local。
- 管理系(`reports`, `announcements`, `navigation_items`, `admin_audit_logs`)：staff/admin のみ。

## 9. 法務・運用上の必須条件（ショップ）

オンラインショップは日本のEC法務を意識して設計する。

### 必須表示
販売価格 / 送料 / 追加費用 / 支払方法と時期 / 引渡時期 / 返品・キャンセル条件 / 事業者情報 / 問い合わせ先

### 実装上の注意
- **無在庫販売であることを誤解させない**（「在庫あり」誤認の禁止）。
- **即納表現は禁止**。納期は具体表現（例：「受注確定後 ○営業日以内に発送」）。
- 返品条件は見つけやすい位置に表示。
- カード情報は自社で保持しない（PSP前提）。
- 法務文言はUIの一部として自然に組み込む（隅に追いやらない）。

3. 通報された投稿を運営が非公開にできる。
4. 世界観・LINE CTA・法務前提の文言が崩れていない。

## 12. 引き継ぎ・開発時の前提

1. 既存「IWASAWA SURF BASE」構想の延長線上で進化させる（新規に作り変えない）。
2. 世界観・導線・運用思想・法務前提を常に維持する。
3. 開発は1ステップずつ進め、動作確認後に次へ（セクション11）。
4. 権限制御は UI + API + RLS の三重化。フロントだけで隠さない。
5. スマホ前提でUI最適化。LINE CTAは強い導線として残す。
6. 管理画面は装飾過多を避け、実務的に見やすく。
7. ショップは無在庫・PSP前提・法務文言をUIに自然に組み込む（Phase2）。
8. 次に拡張しやすい疎結合な構造で組む。
9. 提案・実装の出力は毎回「目的 / 既存構成との関係 / 画面構成 / 主要機能 / 必要データ定義 / UI・UXポイント / 次フェーズ拡張案」の形式で行う。
