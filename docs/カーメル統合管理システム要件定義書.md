# 📘 カーメル統合管理システム 要件定義書

> 株式会社カーメル（carmelonline.jp）の中古車販売業務（ローン販売・車買取・自社リース）を、申込〜納車・返済・アフターサポートまで一元管理する WordPress 統合システムの要件定義。
> マスタープロンプト v1.6 を実装可能な単位に整理し、決定事項を反映してブラッシュアップしたもの。

- **要件定義書バージョン**：v1.6
- **最終更新**：2026-06-16
- **対象システム仕様**：v1.6
- **開発体制**：自社開発（外注なし・AIサポートで実装）
- **インフラ**：WordPress on WPX（既存ドメイン `carmelonline.jp` に同一ドメインで追加）

---

## 0. 確定した重要方針（v1.1で反映）

| # | 論点 | 決定 |
|---|------|------|
| A | GAS版 CarLoan_System との関係 | **WPが正（マスター）**。`carmel_deal` を正データとし、GASは「書類生成・Asanaタスク・LINE通知・AIスコアリング」の**連携専用**として呼び出す |
| B | WPが管理する業務範囲 | **ローン販売・車買取・自社リースの3業務すべて** |
| C | Square決済の対象 | 申込金（手付金）・保証プラン費用・オプション費用・**加盟店の販促購入費**・**会費**。※**車両本体代金はSquare対象外**（ローンは信販会社経由のため） |
| D | 顧客アカウント発行 | **申込フォーム送信時に `customer` ロールを自動発行**し、ログイン情報をLINE/メールで送付 |

> 上記により、WP と GAS の**二重管理を回避**。WPで案件を作成・更新し、必要時にGASへデータを渡して書類・通知・スコアを取得する片方向主導（WP→GAS）を基本とする。

## 2. システム関係図（WP＝正・GAS＝連携）

```
                  ┌──────────────────────────────┐
   申込フォーム ──▶│  WordPress（carmelonline.jp）  │◀── 加盟店/本部の操作
 （自動で顧客発行）  │  ── 正データ：carmel_deal 等 ──│
                  └──────────────┬───────────────┘
        wp_remote_post（Secret Token認証）│ 案件データ送信
                                  ▼
                  ┌──────────────────────────────┐
                  │  GAS（連携エンジン・現行継続）   │
                  │  ・書類生成（Googleドキュメント→PDF→Drive）
                  │  ・AIスコアリング（OpenRouter）
                  │  ・Asanaタスク自動生成
                  │  ・プロライン経由 LINE通知
                  └──────────────┬───────────────┘
                       結果（PDF URL / スコア / task_id）を
                       WPへ返却 or webhookで書き戻し
                                  ▼
                       carmel_deal / carmel_document を更新
```

**外部サービス連携**：Square（決済）／ Google Maps API（陸送費）／ マネーフォワード契約（本部のみ電子署名）／ Notion（学習・外部リンク）／ bbPress（コミュニティ）。
※**在庫はWP内管理（carmel_vehicle）に一本化**。Googleスプレッドシートは正データから除外（必要時はWP→スプシの一方向エクスポートのみ）。

## 5. 業務フローとステータス（業務種別ごと）

業務種別は `carmel_deal` の **`deal_type`**（`loan` / `buyback` / `lease`）で分岐。種別ごとにステータスフローが異なる。

### 5.1 ローン販売（deal_type=loan）

```
仮申込 → AI事前スコアリング（GAS自動） → 信販審査（本部手動）
  → 審査OK / 審査NG
  →（OK時）加盟店マッチング → 書類準備中 → 契約完了（マネーフォワード契約：本部のみ）
  → 納車準備中（陸送手配・費用確定） → 納車済 → 返済管理 → アフターサポート中 → 案件クローズ
```

| 表示名 | slug | PHASE | 主担当 |
|--------|------|:----:|------|
| 仮申込 | `provisional` | 1 | 自動 |
| AIスコア済 | `scored` | 2 | GAS自動 |
| 信販審査中 | `screening` | 2 | 本部 |
| 審査OK | `approved` | 3 | 本部 |
| 審査NG | `rejected` | 3 | 本部 |
| 加盟店マッチング | `matched` | 3 | 本部 |
| 書類準備中 | `doc_prep` | 4 | 加盟店 |
| 契約完了 | `contracted` | 4 | 本部（署名） |
| 納車準備中 | `delivery_prep` | 5 | 加盟店 |
| 納車済 | `delivered` | 6 | 加盟店 |
| アフターサポート中 | `after_support` | 7 | 加盟店/本部 |
| 案件クローズ | `closed` | — | — |

### 5.2 車買取（deal_type=buyback）

```
査定申込 → 査定中 → 査定額提示 → 成約 / 不成立
  →（成約時）書類準備中 → 入金処理（Square：申込金等は対象外、本体は別途）→ 引取完了 → クローズ
```

| 表示名 | slug |
|--------|------|
| 査定申込 | `appraisal_request` |
| 査定中 | `appraising` |
| 査定額提示 | `quoted` |
| 成約 | `bb_agreed` |
| 不成立 | `bb_declined` |
| 書類準備中 | `bb_doc_prep` |
| 引取完了 | `bb_collected` |
| クローズ | `bb_closed` |

### 5.3 自社リース（deal_type=lease）

```
リース申込 → 審査（本部） → 契約 → 納車 → リース中（月々支払・延滞管理・GPS管理）
  → 満了/完済 → クローズ
```

| 表示名 | slug |
|--------|------|
| リース申込 | `lease_request` |
| リース審査 | `lease_screening` |
| 契約完了 | `lease_contracted` |
| 納車済 | `lease_delivered` |
| リース中 | `lease_active` |
| 満了/完済 | `lease_completed` |
| クローズ | `lease_closed` |

### 5.4 ステータス遷移時の自動処理（共通）

- WordPress → GAS webhook → **Asanaタスク自動生成**
- WordPress → プロライン → **顧客にLINE通知**
- WordPress → メール → **関係者に通知**
- ステータスごとに発火する処理は**ステートマシン**として定義（§5.5）

### 5.5 遷移ルール（ステートマシン・抜粋）

| 遷移 | 実行可ロール | 付随する自動処理 |
|------|-------------|-----------------|
| `provisional → scored` | システム（GAS結果） | スコアをWPへ書き戻し |
| `scored → screening` | hq_admin | — |
| `screening → approved/rejected` | hq_admin | 顧客へLINE（審査結果テンプレ） |
| `approved → matched` | hq_admin | 加盟店へ案件アサイン通知 |
| `matched → doc_prep` | 加盟店 | 申込書PDF生成（GAS） |
| `doc_prep → contracted` | **hq_admin のみ** | マネーフォワード契約送付 |
| `delivery_prep` 進入時 | 加盟店 | 陸送費自動計算（Maps API） |
| `delivered → after_support` | 加盟店 | 保証書発行・アフター登録 |

> 権限外ロールが遷移を実行しようとした場合は 403。`customer` はあらゆる遷移操作不可（閲覧のみ）。

## 7. 各ポータルの機能要件

### 7.1 ユーザーマイページ（/mypage）

案件ステータス連動の **PHASE表示制御**（ローン案件基準。買取・リースは簡易版を別途）。

| PHASE | 状態 | 表示内容 |
|-------|------|---------|
| 1 | 仮申込完了 | 申込内容確認・必要書類チェックリスト・**書類アップロード**（ショートコード `[carmel_upload]`・保護保存＋権限付きDL） |
| 2 | 審査中 | 審査進捗バー・待機メッセージ |
| 3 | 審査結果 | OK：次のステップ案内 / NG：理由表示・再申込ボタン |
| 4 | 契約・書類 | 書類確認・署名待ち表示（マネーフォワード契約誘導） |

## 9. 通知設計（オーケストレーター方式）

### 9.0 設計方針

**WordPress を単一の発火点（Notification Orchestrator）とし、チャネルを直接呼ばない。**
各機能はイベントを発火するだけ → オーケストレーターがルーティング表に従って対象者ごとに適切なチャネルへ振り分ける。これにより「同じ通知を複数チャネルから二重送信」「コードのあちこちで個別にAPIを叩く」状態を解消する。

```
[イベント発火]                          ┌─ プロライン Adapter → 顧客LINE
 ステータス変更 / 期日(Cron) /          │
 決済完了 / 審査結果 など   ─────▶  ┌──────────────┐
                                    │ 通知オーケスト │─┼─ LINE WORKS Adapter → 社内
            ルーティング表           │ レーター(WP)   │ │
   （イベント×対象ロール→チャネル）  │ 重複排除/再送  │─┼─ Slack Adapter → 運用/開発
                                    │ ログ/フォール   │ │
                                    │ バック判定     │─┼─ メール Adapter → 正式/控え
                                    └──────────────┘ │   （LINE失敗時フォールバック）
                                            │ 全送信を carmel_notification_log に記録
```

### 9.1 チャネルの役割分担（仕分け）

| チャネル | 主対象 | 種別 | 主な用途 |
|---------|-------|------|---------|
| **プロライン**（LINE） | 顧客 `customer` | B2C・進捗/販促 | 申込受付・審査結果・契約/署名・納車・返済リマインダー・車検/保険案内 |
| **LINE WORKS** | 社内（本部・加盟店スタッフ） | B2B・業務連絡 | 案件アサイン・対応依頼・社内エスカレーション・期限間近アラート |
| **Slack** | 本部運用・開発 | システム監視 | 連携失敗（GAS/Square/Maps）・エラー・延滞検知サマリ・日次/週次KPI |
| **メール** | 顧客＋社内 | 正式記録・フォールバック | 正式通知・書類添付・**LINE未連携/失敗時の自動フォールバック**・監査控え |

> Asanaタスク生成はGAS連携（§8.1）として継続。LINE WORKS通知と役割が重複しないよう、Asana＝タスク管理、LINE WORKS＝即時連絡 と整理。

### 9.2 ルーティング表（イベント × 対象 → チャネル）

| トリガー | 顧客（プロライン） | 社内（LINE WORKS） | システム（Slack） | メール |
|---------|:---:|:---:|:---:|:---:|
| 仮申込受付＋アカウント発行 | ✅ | — | — | ✅(控え) |
| AIスコア完了 | — | ✅(本部) | — | — |
| 審査結果（OK/NG） | ✅ | ✅(加盟店) | — | ✅(OK時) |
| 加盟店アサイン | — | ✅(加盟店) | — | ✅ |
| 契約・署名依頼 | ✅ | — | — | ✅ |
| 納車日確定 | ✅ | ✅(加盟店) | — | — |
| 決済完了/失敗（Square） | ✅(完了) | ✅(失敗) | ✅(失敗) | ✅ |
| 返済リマインダー（3日前/前日/当日） | ✅ | — | — | F |
| 延滞督促（1日/5日/14日超） | ✅ | ✅(本部) | ✅(サマリ) | F |
| 車検案内（90/60/30日前） | ✅ | ✅(加盟店) | — | F |
| 保険更新案内（90/30日前） | ✅ | ✅(加盟店) | — | F |
| 整備/保証案内 | ✅ | — | — | — |
| 連携失敗・システムエラー | — | — | ✅ | — |

凡例：✅＝送信、F＝LINE未連携/失敗時のフォールバック、—＝送信なし。

### 9.3 オーケストレーターの仕組み

- **チャネルアダプタ**：`proline_adapter` / `lineworks_adapter` / `slack_adapter` / `mail_adapter` を共通インターフェースで実装。チャネル追加時はアダプタ追加のみ
- **フォールバック**：顧客宛LINE（プロライン）が**未連携 or 送信失敗**の場合、同一テンプレートを**メールへ自動再送**（決定事項）
- **重複排除（冪等性）**：`event_id × recipient × channel` をユニークキーとし、同一イベントの二重送信を防止
- **リトライ**：送信失敗は指数バックオフで再試行。最終失敗は **Slackへ運用アラート**
- **配信ログ**：全送信を `carmel_notification_log`（§6.9）に記録（日時・チャネル・宛先・結果・リトライ回数）
- **テンプレート管理**：本部が /hq でチャネル別テンプレートを編集（差込変数：氏名・案件番号・期日・金額等）
- **オプトアウト**：顧客は /mypage で販促系（任意）の受信可否を設定。**取引上必須の通知（審査・契約・返済・期日）はオプトアウト対象外**
- **静音時間**：顧客向け配信は深夜帯を避ける送信ウィンドウを設定可能

### 9.4 定期処理（WP-Cron）

| 時刻 | 処理 → 発火するイベント |
|------|------|
| 毎朝 | 返済リマインダー／延滞検知・利息計算／車検・保険 満了判定（90/60/30日前）／定期点検案内 → オーケストレーターへ |
| 毎週月曜 | 週次レポート生成 → Slack(KPI)＋メール(本部) |

> 現行GASトリガーと重複しないよう、**返済・延滞・アフターの定期処理はWP-Cronに一本化**（WPが正のため）。GASは書類/スコア/Asana連携に限定。

## 10. セキュリティ・個人情報

- 各ページ（/mypage・/store・/hq）は**ログイン必須**。権限外URL直アクセスは **403 or /login リダイレクト**
- 案件データは `customer_id` または `store_id` で**必ずフィルター**（他者データを絶対に見せない）
- GAS Webアプリは **Secret Token認証**、Square webhookは **署名検証必須**、WPフォームは **nonce必須**
- **監査ログ**：ステータス変更・書類生成・マネーフォワード契約送付・審査結果入力を記録（誰が・いつ・何を）。※ステータス変更は `Carmel_Deal_Status` が `_carmel_status_history`（from/to・実行者・日時）に記録済み
- **個人情報保護**：本人確認書類・収入証明等の機微情報はアクセス制御を厳格化。**書類保存期間ポリシー**を§10.1に定義
- 顧客アップロード書類は権限者のみ閲覧可（直URL推測防止）

### 10.1 書類保存期間ポリシー（確定）

| 区分 | 保存期間 | 期間後の扱い |
|------|---------|-------------|
| 契約書・売買関連書類 | 案件クローズ後 **7年**（税務・帳簿保存年数に整合） | 本部の手動確認のうえ削除 |
| 本人確認書類・収入証明（成約） | 案件クローズ後 **7年** | 同上 |
| 本人確認書類等（審査NG/不採用） | 確定後 **6か月** | 削除 |
| 監査ログ（`_carmel_status_history`） | 保持（バックアップ対象） | — |

- **自動破棄はしない**（誤削除・法令対応のリスク回避）。保存期間を超過した書類は**本部へ通知で警告**し、担当者が確認のうえ削除する運用。
- 顧客から削除請求があった場合は、法定保存義務のある書類を除き対応（個人情報保護法）。

3. 加盟店の電子署名は **加盟店独自の外部サービス**で対応（システム外）
4. 学習コンテンツは **Notion**で管理（WordPress内に実装しない）
5. Asanaは **現行GAS自動化を継続**（落ち着いたらWP移行）
6. **サブドメイン不使用**（carmelonline.jp に同一ドメインで追加）
7. 保証プランは固定価格ではなく **価格表PDF方式**
8. 在庫管理は **WP内で一元管理（carmel_vehicle）**。全操作をWPで完結・反映。スプレッドシートは正データから除外（必要時はWP→スプシの一方向エクスポートのみ）
9. データの正は **WP（carmel_deal）**。GASは連携専用で二重管理しない

## 15. 用語集

| 用語 | 意味 |
|------|------|
| 案件（deal） | 1件の取引単位。`carmel_deal` で管理。`deal_type` で loan/buyback/lease に分岐 |
| 本部 | カーメル運営本部（hq_admin）。窓口一本化・審査・署名を担当 |
| 加盟店 | フランチャイズ店舗（store_owner/store_staff） |
| 信販会社 | ローン審査を行う与信会社（C001オリコ〜C008自社ローン） |
| PHASE | /mypage で顧客に見せる段階表示（1〜7） |
