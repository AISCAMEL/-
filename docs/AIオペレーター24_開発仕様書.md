## 2. 技術構成

```
顧客が電話 → Twilio Phone Number → Twilio Voice / Conversation Relay
  → 自社 WebSocket Gateway → AI Orchestrator → LLM / FAQ / DB / 通知 / 管理画面
```

| レイヤ | MVP採用 | 候補 |
|--------|---------|------|
| Frontend | Next.js + React + Tailwind | — |
| Backend | Node.js + TypeScript (Fastify) | NestJS / Python FastAPI |
| DB | PostgreSQL / Supabase | — |
| 電話基盤 | Twilio Voice + Conversation Relay | — |
| AI | **OpenAI（MVP優先）** | Claude / Gemini |
| 通知 | Resend（メール）+ Slack Webhook | SendGrid |
| 認証 | Supabase Auth | Clerk / Auth0 |
| ホスティング | Vercel(front) + Render/Railway(back) | AWS / GCP |

## 3. Twilio方針

- 裏側の通信基盤として利用。顧客にTwilioを直接操作させず、自社管理画面から設定。
- **MVP**：1つまたは少数のTwilio番号でデモ・導入。
- **将来拡張**：Twilio Subaccount で顧客ごとに番号・通話・利用量・録音・請求を分離。
- Conversation Relay 採用理由：STT/TTS/低遅延会話/WebSocket連携 がAI音声エージェントに最適。

TwiML例・WSメッセージ仕様は `api.md` を参照。

## 8. AI会話仕様

詳細は **`ai-conversation.md`**。要点：
- FAQ外は推測しない。予約は「予約希望として受付」と表現。禁止事項を厳守。
- ターン毎にJSON出力（reply/intent/state/extracted/should_transfer/should_end_call/next_action/confidence）。
- 通話後に要約JSONを生成し、管理画面・通知に利用。

---

## 9〜13. 業務ルール（要約）

- **予約**：希望日時→名前→内容→確認。確定とは言わない。
- **問い合わせ**：FAQ該当は回答、なければ折り返し・担当者確認へ。
- **折り返し**：名前→会社名→要件→番号確認（発信者番号を確認）。
- **転送**：人希望/クレーム/緊急/契約・専門判断/AIが確信なし/同一質問3回以上 → `should_transfer=true`。転送先なしは折り返しへ。
- **クレーム**：謝意→簡単な聞き取り→人間へ。AI単独で完結させない。

詳細・例文は `ai-conversation.md` 第5章。

---

## 14〜15. 要約・通知

- 要約：ログにある情報のみ、不明はnull、短文。出力JSONは `ai-conversation.md` 第6章。
- 通知：件名「【AIオペレーター24】新しい電話受付がありました」。本文＝顧客名/会社名/電話番号/要件/内容/希望日時/対応結果/次の対応/通話要約。
