# 📥 請求書受領・自動仕訳システム 設計書 v1.1

> 各業者からメールで届く請求書を、**既存のGmailラベル運用を起点**に、受領 → 抽出 → 保存 → 分類 → 台帳管理 → 承認 → 仕訳まで一気通貫で自動化するシステムの設計書。
> GAS（Google Apps Script）＋ Googleスプレッドシート＋ Googleドライブ＋ AI/OCR ＋ Asana ＋ マネーフォワード を連携する。

最終更新：2026-06-07
対象読者：経営者・経理担当（非エンジニア）＋ 実装担当エンジニア

> **v1.1 追記**：本案件の背骨「Gmail請求書の仕分け → マネーフォワード」を §14 として確定版に整理し、`templates/` に業者マスタ・仕訳パターン等の雛形を同梱しました。まずは §14 と `templates/README.md` を参照してください。

## 1. 現状理解

| 項目 | 現状 |
|------|------|
| 請求書の届き方 | 各業者から**メールで**届く |
| 添付形式 | PDF・画像（JPG/PNG/スクショ）・**本文に金額直接記載**の3パターンが混在 |
| 仕分け方法 | 受信メールを**手作業でGmailラベル分け**して管理 |
| フォーマット | **業者ごとにバラバラ**（決まった様式がない） |
| 後工程 | 台帳記録・支払管理・会計仕訳が手作業（推定） |
| 会計ソフト | マネーフォワード（ログイン・設定・仕訳運用の支援も必要） |

**現状の課題（仮説）**
1. ラベル分けは手作業 → 抜け漏れ・属人化のリスク。
2. 請求書の金額・支払期限を人が目視で台帳転記 → 転記ミス・支払漏れ。
3. 添付ファイルがメールに埋もれる → 後から探せない（証憑管理が弱い）。
4. 会計仕訳が手入力 → 月末に負荷集中。

> 💡 ポイント：**「ラベルは活かす」**が大前提。ラベルを壊さず、ラベルを"処理の入口（トリガー）"として使う設計にします。

## 2. 目的整理

### 達成したいこと（優先度順に再整理）

| # | 目的 | 何が嬉しいか |
|---|------|------------|
| ① | ラベル付きメールを自動監視 | 人が「処理対象」を探さなくてよくなる |
| ② | 本文・PDF・画像から請求書情報を抽出 | 目視転記が不要に |
| ③ | スプレッドシート台帳へ自動記録 | 一覧・検索・集計が一瞬でできる |
| ④ | 添付をDriveへ業者別・年月別に自動保存 | 証憑がいつでも探せる（監査対応） |
| ⑤ | Asanaで確認・承認タスクを自動作成 | 「誰が・いつ承認したか」が残る |
| ⑥ | マネーフォワードへ仕訳連携 | 月末の入力作業を削減 |
| ⑦ | 将来：仕訳候補のAI自動提案＋人の確認 | 経理判断を支援・標準化 |

### 設計方針（スタンス）
- **完全自動を狙わない。** 抽出と転記は自動化、**金額確定・仕訳承認は人が1クリックで確認**する「半自動（Human-in-the-loop）」を基本にする。
- **証憑（しょうひょう）＝請求書の原本ファイル**は必ず元データのまま保存し、AI抽出値とは分けて管理する（後で照合できるように）。
- まずMVPで「①〜④（受領〜台帳〜保存）」を回し、効果を確認してから⑤⑥⑦を足す。

## 3. 全体システム構成案

### 3-1. データの流れ（俯瞰図）

```
 ① 業者              ② Gmail               ③ GAS（自動化エンジン）
┌────────┐      ┌──────────────┐      ┌──────────────────────────┐
│ 請求書  │ mail │ 受信トレイ     │      │ a. ラベル付きメール検索     │
│ メール  │─────▶│ ＋ 手動ラベル  │─────▶│ b. 添付・本文の取得         │
└────────┘      │   （既存運用） │      │ c. AI/OCRで項目抽出         │
                └──────────────┘      │ d. Driveへ証憑保存          │
                                       │ e. Sheetへ台帳記録          │
                                       │ f. Asanaへ承認タスク作成     │
                                       │ g. MF連携データ生成          │
                                       └───────┬──────────────────┘
              ┌────────────────────┬───────────┼───────────────┐
              ▼                    ▼            ▼               ▼
        ④ Google Drive      ⑤ Sheet台帳   ⑥ Asana        ⑦ マネーフォワード
        （証憑PDF/画像）     （請求一覧）   （承認タスク）  （仕訳・会計）
        業者別/年月別        ステータス管理  承認→完了      CSV or API
```

### 3-2. 各コンポーネントの役割

| 層 | ツール | 役割 | なぜこれを使うか |
|----|--------|------|----------------|
| 受領 | **Gmail** | 請求書メールの受信・既存ラベル | 既存運用をそのまま活かせる |
| 制御 | **GAS** | 全処理のオーケストレーション（指揮） | 無料・Googleと密連携・時間トリガーで定期実行 |
| 抽出 | **AI/OCR** | PDF/画像/本文から項目を構造化 | 業者ごとにフォーマットが違っても対応可 |
| 台帳 | **Googleスプレッドシート** | 請求一覧・ステータス・集計 | 経理が直接見て・直せる |
| 証憑 | **Googleドライブ** | 原本ファイルの保管庫 | フォルダ階層で監査対応 |
| 承認 | **Asana** | 確認・承認タスクのワークフロー | 担当・期限・履歴が残る |
| 会計 | **マネーフォワード** | 仕訳・支払・会計処理 | 既存の会計基盤 |

> *オーケストレーション*：複数の処理を「正しい順番で・失敗したら止める」ように束ねて実行すること。GASがその司令塔になります。

### 3-3. AI/OCR の選択肢（比較）

| 方式 | 内容 | 精度 | コスト | 推奨 |
|------|------|------|--------|------|
| **A. AIマルチモーダル**（Claude / Gemini にPDF・画像を直接渡す） | 1回のAPIで「読む＋項目抽出」まで | ◎ 高（レイアウト崩れに強い） | 従量（1通あたり数円〜） | **本命** |
| B. OCR専用（Google Cloud Vision / Document AID等）→ AIで整形 | テキスト化してから抽出 | ○ | OCR＋AIの二重 | 大量・低コスト重視時 |
| C. ルールベース（正規表現） | 業者別に金額位置を決め打ち | △（様式変更に弱い） | 0円 | 定型業者のみ補助的に |

> 推奨：**A方式（AIマルチモーダル）を主軸**にする。既存の `OpenRouter` 経由（社内に実績あり）で Claude / Gemini を呼べば、PDFも画像も本文も同じ仕組みで処理できます。

## 4. MVP（最小実装）案

> **狙い：2〜3週間で「受領→台帳→証憑保存」までを回し、手作業の転記をゼロにする。**
> 承認(Asana)・会計連携(MF)はまだ入れず、まず"見える化"を取りに行く。

### 4-1. MVPスコープ

| 含む ✅ | 含まない ❌（次フェーズ） |
|---------|------------------------|
| 指定ラベルのメール監視（時間トリガー） | Asana承認タスク |
| 添付PDF/画像のDrive保存（業者別/年月別） | マネーフォワード連携 |
| AIで主要項目を抽出 | 仕訳候補の自動提案 |
| スプレッドシート台帳へ自動記録 | 複雑な例外の自動リカバリ |
| 処理済みフラグ（重複防止） | 多通貨・外貨対応 |
| 抽出失敗時に「要確認」マーク | |

### 4-2. MVPの動き（1サイクル）

1. 1時間ごとにGASが起動。
2. ラベル `請求書/未処理` が付いた未処理メールを検索。
3. 添付（PDF/画像）を取得。なければ本文テキストを使う。
4. AIに渡して「請求元・請求日・金額・税額・支払期限・摘要」を**JSONで**受け取る。
5. 添付を Drive `/請求書/{業者名}/{YYYY}/{YYYYMM}/` に保存。
6. 台帳シートに1行追記（リンク・抽出値・信頼度）。
7. 抽出が曖昧なら `ステータス=要確認` にする。
8. メールのラベルを `請求書/未処理` → `請求書/処理済` に張り替え（＝二重処理防止）。

### 4-3. MVPで用意するもの（チェックリスト）
- [ ] Gmailラベル：`請求書/未処理`・`請求書/処理済`・`請求書/エラー`（既存ラベル名でも可）
- [ ] 台帳スプレッドシート（列設計は §7）
- [ ] Driveルートフォルダ `請求書`
- [ ] AI APIキー（OpenRouter等）
- [ ] GASプロジェクト（スクリプトプロパティに各種ID/キーを保存）
- [ ] 時間トリガー（1時間ごと）

## 6. 必要な連携先 / API 一覧

| 連携先 | 用途 | 接続方式 | 認証 | 備考 |
|--------|------|----------|------|------|
| **Gmail** | メール監視・ラベル操作・添付取得 | GAS標準 `GmailApp` | Googleアカウント（OAuth自動） | 追加コスト無し |
| **Googleスプレッドシート** | 台帳・マスタ・ログ | GAS標準 `SpreadsheetApp` | 同上 | 同上 |
| **Googleドライブ** | 証憑保存 | GAS標準 `DriveApp` | 同上 | 同上 |
| **AI（抽出）** | OpenRouter (Claude/Gemini) | `UrlFetchApp` でREST | APIキー | 従量課金 |
| **OCR（任意）** | Google Cloud Vision / Document AI | REST | APIキー/サービスアカウント | A方式なら原則不要 |
| **Asana** | 承認タスク | REST API v1 | Personal Access Token | 無料枠で可 |
| **マネーフォワード クラウド会計** | 仕訳連携 | ①CSVインポート ②API | ①不要 ②OAuth | §6-1参照 |

### 6-1. マネーフォワード連携方式の比較 ★重要論点

| 方式 | 内容 | 難易度 | 自動度 | 推奨フェーズ |
|------|------|--------|--------|------------|
| **① CSV／中間台帳** | 台帳から「仕訳インポート用CSV」を生成 → MF画面から取込 | 低 | 半自動（取込は人） | **MVP〜初期本番（推奨スタート）** |
| **② MF API（仕訳連携）** | APIで仕訳を直接登録 | 高（要API契約・OAuth・審査） | 高（全自動） | 安定後に移行 |
| **③ MFのメール/連携機能** | MF側の証憑アップロード等を使う | 中 | 中 | 補助的 |

> 推奨：**まず①CSV方式**で「人がMFに取り込む直前の状態」まで自動化。運用が固まり仕訳ルールが安定したら②APIへ。
> ⚠️ MF APIは利用条件・提供範囲が変わりやすいため、**最新の利用可否は確認事項（§13）に記載**。

### 7-4. AIに返させるJSONスキーマ（抽出の契約）

```json
{
  "vendor_name": "string",        // 請求元
  "invoice_number": "string|null",
  "invoice_date": "YYYY-MM-DD|null",
  "due_date": "YYYY-MM-DD|null",
  "subtotal": "number|null",      // 税抜
  "tax": "number|null",
  "total": "number|null",         // 税込・最重要
  "currency": "JPY",
  "description": "string|null",   // 摘要

```

### 7-5. Driveフォルダ命名規則

```
請求書/
 └ {正規化業者名}/
     └ {YYYY}/
         └ {YYYYMM}/
             └ {YYYYMMDD}_{業者名}_{合計金額}_{請求ID}.pdf
```

## 8. 自動化フロー（処理ステップ詳細）

### 8-1. メイン処理（時間トリガー：1時間ごと）

| # | ステップ | GASの処理 | 失敗時 |
|---|----------|-----------|--------|
| 1 | 対象抽出 | `GmailApp.search('label:請求書/未処理')` | ログのみ |
| 2 | 重複チェック | メールIDが台帳C列に既存ならスキップ | スキップ |
| 3 | 添付/本文取得 | 添付PDF/画像を取得。無ければ本文HTML→テキスト | §9-1 |
| 4 | AI抽出 | 添付Blob or 本文をAIへ送りJSON取得 | §9-2 |
| 5 | 正規化 | 業者マスタ照合・金額の数値化・日付整形 | 要確認 |
| 6 | Drive保存 | フォルダ自動作成→ファイル保存→URL取得 | §9-3 |
| 7 | 台帳記録 | 1行追記（信頼度・ステータス込み） | 再試行 |
| 8 | 仕分け | 信頼度<閾値 or 必須欠落→`要確認`、else`承認待ち` | — |
| 9 | ラベル張替 | `未処理`除去→`処理済`付与 | §9-4 |
| 10 | ログ | `処理ログ`へ結果記録 | — |

### 8-2. 承認フロー（本番）

1. ステータス`承認待ち`／`要確認`の行を検出。
2. Asanaに承認タスク作成（タスク名・金額・業者・証憑リンク・期限）。
3. 担当が内容確認 → Asanaでタスク完了 or コメントで「修正依頼」。
4. GASがAsana完了を検知 → 台帳を`承認済`に更新＋承認者/日時記録。
5. 修正依頼の場合は`要確認`へ戻す。

### 8-3. 会計連携フロー（本番）

1. `承認済`かつ`MF連携日`が空の行を抽出。
2. 業者マスタから勘定科目・税区分を補完（無ければAI提案）。
3. MF仕訳インポート用CSVを生成（§6-1①）→ 共有フォルダ/メールで経理へ。
4. 経理がMFへ取込 → 台帳を`仕訳済`・`MF連携日`記録。
（API化後は3-4を自動化）

## 9. 例外処理・エラー時対応

| ケース | 検知 | 自動対応 | 人の対応 |
|--------|------|----------|----------|
| 9-1 添付も本文も読めない | 添付0件＋本文短すぎ | `エラー`＋`請求書/エラー`ラベル | 手動で内容入力 |
| 9-2 AI抽出失敗/タイムアウト | API異常・JSON壊れ | 最大3回リトライ（指数バックオフ）→ なお失敗で`要確認` | 値を手入力 |
| 9-3 Drive保存失敗 | 例外 | リトライ→失敗で台帳に「証憑未保存」 | 後で手動添付 |
| 9-4 ラベル張替失敗 | 例外 | 次回再処理されるが、重複チェック(8-1#2)で二重記録は防止 | — |
| 9-5 金額不一致 | 小計＋税≠合計 | `要確認`＋警告 | 目視確認 |
| 9-6 二重請求疑い | 同業者×同金額×近接日 | `要確認`＋「重複候補:INV-xxxx」 | 重複か判断 |
| 9-7 信頼度が低い | confidence<0.8 | `要確認` | 確認・修正 |
| 9-8 想定外の言語/外貨 | currency≠JPY等 | `要確認` | 個別対応 |
| 9-9 API上限/コスト超過 | 月次カウンタ | 処理一時停止＋管理者通知 | 上限見直し |

**共通原則**
- **冪等性（べきとうせい）**：同じメールを何度処理しても結果は1件だけ（メールID重複チェックで担保）。*冪等性＝同じ操作を繰り返しても結果が変わらない性質。*
- **失敗は止めずに隔離**：1通の失敗で全体を止めない。失敗分は`エラー`/`要確認`に隔離して後追いできる。
- **証憑優先**：抽出に失敗しても、原本ファイルだけは必ずDriveに残す。

## 14. 本線設計（確定版）：Gmail請求書 → マネーフォワード

> 承認(Asana)・高度化は枝葉。まず**この1本（受領→仕分け→台帳→仕訳→MF）**を通す。

### 14-1. 背骨フロー
```
Gmailラベル付き請求書 → GAS(抽出+機械検証+Drive保存) → 台帳(仕訳の形まで完成) → マネーフォワード取込
```
台帳に「借方・貸方・勘定科目・税区分」まで作り込み、人が1画面で確認できる状態にする。

### 14-2. ★自作 vs マネーフォワード既存機能（要判断）
要件（受領→OCR→承認→仕訳→支払）の大半はMFの既存サービスにも存在する。

| MF製品 | できること | 要件との重なり |
|--------|-----------|--------------|
| クラウド会計 | 仕訳・帳簿・**仕訳CSVインポート** | 仕訳の最終受け皿（必須） |
| クラウドBox | 証憑保管・電子帳簿保存法対応・AI-OCR | Drive保存と重複 |
| クラウド債務支払 | 受領請求書のAI-OCR・承認・仕訳・支払 | **要件のほぼ全部** |

**結論**：
- MF「債務支払/Box」を使えるなら、**GASは"Gmail→MFへの自動投入役"に徹する**（抽出を自作しない）＝最小工数。
- 使わない/コスト重視なら、**GASで抽出まで作りMF会計へ仕訳CSV**を流す（§4 AI精度設計が活きる）。
- 判断は契約状況とコスト次第 → 確認事項#7・新#13-A。

### 14-3. ルート比較
| | A: GAS抽出→MF会計CSV | A': GAS投入→MF債務支払/Box | B: GAS→MF API |
|---|---|---|---|
| 抽出/OCR | 自作(GASのAI) | **MFに任せる** | 自作 |
| 承認 | Sheet/Asana | **MF内ワークフロー** | Sheet/Asana |
| 月額 | 安(AI従量のみ) | MF債務支払料金 | 安 |
| 工数 | 中 | **小** | 大 |
| 電帳法 | Drive自前 | **MF Box標準** | 自前 |
| 推奨 | コスト最優先(スタート) | MF契約が手厚いなら最有力 | 安定後の上級 |

### 14-4. 仕訳マッピング（自動化のキモ）
**業者マスタに「この業者＝この勘定科目・税区分・仕訳パターン」を1回登録すれば、以後の仕訳は全自動で決まる。**

| 仕訳項目 | 取得元 |
|----------|--------|
| 取引日 | 台帳 請求日 |
| 借方 勘定科目/税区分 | 業者マスタ既定値（無ければAI提案） |
| 借方 金額・仮払消費税 | 抽出値（税抜/税込はMF設定に合わせる） |
| 貸方 勘定科目 | 未払金 or 買掛金（業者マスタ） |
| 貸方 金額 | 合計（税込） |
| 摘要 | 業者名＋摘要＋請求書番号 |
| 源泉 | パターン`GENSEN`なら預り金行を追加 |

仕訳の型は `templates/仕訳パターン明細.csv` で定義（標準/税込/源泉あり/軽減税率混在）。

### 14-5. MF仕訳インポートCSV（注意）
⚠️ **正確な列名・順序・税区分表記はMFの製品/版で異なる**。実装前に**MF管理画面から仕訳インポート用テンプレCSVを1つDL**し、それに出力を合わせる（確認事項#13-A）。同梱雛形はMF取込の「手前＝社内台帳の完成形」。
